# System Design Visualizer 🚀

**Learn system design by doing** — study canonical cloud & distributed system architectures, practice AI-evaluated architectural trade-offs, and master weak areas using spaced repetition.

---

## 📌 Why We Are Building This

### The Problem with Traditional System Design Prep
Traditional system design interview preparation has a critical flaw: **candidates memorize architecture diagrams without understanding the underlying trade-offs.** 

When asked *"Why did you pick Cassandra over PostgreSQL?"* or *"What fails when write traffic spikes 100x?"*, memorization breaks down because active architectural reasoning was never practiced.

### The Solution
**System Design Visualizer** bridges this gap by turning static design reading into an **adaptive, interactive learning feedback loop**:
1. **Interactive Visualization**: Render dynamic Mermaid.js diagrams alongside hand-crafted architectural trade-offs.
2. **AI-Driven Trade-Off Scenarios**: Solve real-world architectural challenge prompts generated by Groq (LLM).
3. **Multi-Dimensional AI Evaluation**: Receive objective, structured feedback grading your answer across **4 key dimensions**: *Correctness*, *Trade-off Reasoning*, *Scalability Awareness*, and *Missed Points*.
4. **SM-2 Spaced Repetition**: Never forget concepts — review intervals adaptively adjust based on your answer quality scores.
5. **Analytics & Weak Spot Tracking**: Pinpoint exactly where your knowledge gap lies (e.g., database scaling vs. cache invalidation).

## 🏗 Architecture Principles

- **Content First, Zero-Code Extensibility**: System design topics are defined as structured JSON schemas in `/content/`. Adding a new topic is zero-code: place a `.json` file in `/content/` and seed.
- **Server Components by Default**: Data loading occurs on the server to reduce bundle sizes; Client Components (`'use client'`) are reserved for interactive state (Zustand, Mermaid rendering, forms).
- **Decoupled Business Logic**: API routes (`app/api/*`) are thin controllers that delegate logic to clean service classes (`lib/services/*`).
- **Structured AI Outputs**: AI evaluations bypass unstructured text outputs by utilizing JSON schema validation to guarantee 4-dimensional feedback.

## 📚 Curriculum (Pre-built Topics)

The application launches with **8 canonical system design topics**:

1. **URL Shortener** (`url-shortener.json`): Base62 vs. Hash truncation, key generation service, cache strategy.
2. **Rate Limiter** (`rate-limiter.json`): Token Bucket vs. Leaky Bucket vs. Sliding Window Log, Redis integration.
3. **Distributed Cache** (`distributed-cache.json`): Consistent Hashing, Cache-Aside vs. Write-Through, Eviction policies (LRU/LFU).
4. **Chat Application** (`chat-application.json`): WebSockets vs. Long Polling, message ordering, push notifications.
5. **Video Streaming** (`video-streaming.json`): Transcoding pipelines, HLS vs. DASH, CDN chunking strategies.
6. **Search Autocomplete** (`search-autocomplete.json`): Trie data structure, Redis zset, prefix aggregation workers.
7. **News Feed** (`news-feed.json`): Fan-out on Write vs. Fan-out on Read, hybrid push-pull models.
8. **Job Queue & Worker System** (`job-queue.json`): At-least-once delivery, dead-letter queues, idempotent processing.

## 🚀 Getting Started

### Prerequisites
- **Node.js**: `v18.x` or higher
- **npm**: `v9.x` or higher
- **MongoDB**: Local instance or MongoDB Atlas URI
- **Groq API Key**: For AI quiz generation & structured evaluation

### Local Setup

1. **Clone the repository:**
   ```bash
   git clone https://github.com/your-username/SystemDesignVisualizer.git
   cd SystemDesignVisualizer
   ```

2. **Install dependencies:**
   ```bash
   npm install
   ```

3. **Configure environment variables:**
   Create a `.env.local` file in the root directory:
   ```env
   MONGODB_URI=mongodb://localhost:27017/system-design-visualizer
   GROQ_API_KEY=your_groq_api_key_here
   ```

4. **Seed Content Topics into Database:**
   ```bash
   npm run seed
   ```

5. **Run the Development Server:**
   ```bash
   npm run dev
   ```
   Open [http://localhost:3000](http://localhost:3000) in your browser.

## 💡 Build Approach & Methodology

We adopt the **Skateboard Methodology** — shipping the thinnest usable slice end-to-end first, then iteratively growing functionality. Development is managed using structured specs stored under `docs/specs/` and backed by dedicated agent workflows (`/architect`, `/develop`, `/check`, `/test`).
